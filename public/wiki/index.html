Deployment handbook

Felixx Identity on Vercel, plus integration patterns for other services

The repository is split into three concerns: the Express auth server, the account portal, and any external websites that use the same central login. This guide explains where each piece runs, how to deploy the auth server to Vercel, and how to add new clients without copying brittle auth code everywhere.

Auth server Express app, exported as a Vercel Function
Database PocketBase at pocketbase.felixx.dev
Clients Authorization Code + PKCE everywhere

Architecture

What lives where

  • server/src/app.js: OAuth endpoints, account APIs, and the Express app export.
  • pocketbase/collections.schema.json: the collections the auth layer expects.
  • apps/accounts/: first-party account portal.
  • apps/example-client/: a minimal client that demonstrates a reusable OAuth integration.
  • public/wiki/index.html: this guide page, served at /wiki.

Important separation

  • PocketBase is not the auth server. It stores records and user accounts.
  • The Express app is the auth server. It signs tokens and enforces OAuth rules.
  • Other websites are clients. They redirect users to Felixx Identity and exchange codes for tokens.

Deploying The Auth Server On Vercel

Vercel's Express guidance says the app can be deployed with zero configuration when the Express module is exported as a default export, and that static assets should live in public/**.

1. Export the app

The Express app is exported from server/src/app.js. The file only calls app.listen when run directly, so Vercel can import it as a function while local development still works.

2. Keep the runtime Node-based

The app depends on Node APIs and the PocketBase SDK. Keep it in the Node.js runtime and avoid moving auth logic to Edge unless you rewrite the dependencies for that environment.

3. Attach environment variables

Use Vercel environment variables for all secrets and environment-specific settings. New values only affect new deployments.

4. Use a custom domain

Point the production domain you want to the Vercel project, then set AUTH_ISSUER and any client configs to the same production URL. 

server/src/app.js

if (process.argv[1] && pathToFileURL(process.argv[1]).href === import.meta.url) {
  app.listen(config.port, () => {
    console.log(`Felixx auth server listening on :${config.port}`);
  });
}

export default app;
On Vercel, connect the repository, set the production domain, and push a deployment. The Express app becomes a single Vercel Function, while files in public/** are served statically.

Environment Variables

Variable Purpose Example
POCKETBASE_URL The PocketBase instance the auth server uses for admin reads/writes and user auth. https://pocketbase.felixx.dev
POCKETBASE_ADMIN_EMAIL PocketBase admin account email. fake.felix@protonmail.com
POCKETBASE_ADMIN_PASSWORD Stored in Vercel as a secret environment variable. Set in Vercel dashboard or CLI
ACCESS_TOKEN_SECRET JWT signing secret for access tokens. A long random secret
REFRESH_TOKEN_SECRET JWT signing secret for refresh tokens. A different long random secret
ALLOWED_ORIGINS Comma-separated first-party origins that may call the auth API from browsers. https://accounts.felixx.dev,https://app.example.com
AUTH_ISSUER The public issuer string embedded in JWTs and documentation. https://auth.felixx.dev
ACCOUNTS_APP_URL Where account links should point. https://accounts.felixx.dev
Vercel environment variables are environment-scoped. If you change them, redeploy. Previous deployments do not inherit the new values automatically.

PocketBase Setup

Required collections

  • users
  • oauth_clients
  • oauth_auth_codes
  • oauth_refresh_tokens
  • ai_usage

Admin access

The auth server authenticates to PocketBase as an admin user to manage OAuth clients, authorization codes, refresh tokens, and profile updates. The admin email is fixed in the environment and the password should come from a secret store. 

.env production example

POCKETBASE_URL=https://pocketbase.felixx.dev
POCKETBASE_ADMIN_EMAIL=fake.felix@protonmail.com
POCKETBASE_ADMIN_PASSWORD=...secret...
ACCESS_TOKEN_SECRET=...secret...
REFRESH_TOKEN_SECRET=...secret...
ALLOWED_ORIGINS=https://accounts.felixx.dev,https://new-app.example.com
AUTH_ISSUER=https://auth.felixx.dev
ACCOUNTS_APP_URL=https://accounts.felixx.dev

Integrating Other Services

Standard client flow

  1. Register a client in PocketBase with a unique client_id.
  2. Add the exact redirect URI for that service.
  3. Make the service redirect users to /authorize with PKCE.
  4. Exchange the returned authorization code at /token.
  5. Store only what the service needs, and prefer short-lived access tokens plus refresh token rotation.

Browser-only apps

Use the shared helper module in apps/shared/oauth.js. Inject per-site config in the page HTML and keep the auth server URL and client id out of the app logic.

Backend-rendered apps

The browser can still start the authorization redirect, but the code exchange and refresh flow can also be moved to a backend if you want fewer tokens in the browser.

The important part is to keep Authorization Code + PKCE intact, because it is the flow this repo is already built around.

Example Client Configuration

<!-- apps/example-client/index.html -->
<script>
  window.FELIXX_AUTH_CONFIG = {
    authBaseUrl: "https://auth.felixx.dev",
    clientId: "felixx-example-client",
    scope: "openid profile email offline_access",
    storageKey: "felixx-example-session",
    pkceStateKey: "felixx-example-pending",
    pkceVerifierKey: "felixx-example-code-verifier"
  };
</script>
<!-- apps/accounts/index.html -->
<script>
  window.FELIXX_AUTH_CONFIG = {
    authBaseUrl: "https://auth.felixx.dev",
    clientId: "felixx-accounts",
    scope: "openid profile email offline_access",
    storageKey: "felixx.accounts.oauth",
    pkceStateKey: "felixx.accounts.oauth_state",
    pkceVerifierKey: "felixx.accounts.code_verifier"
  };
</script>

Reusable helper module

The browser helper in apps/shared/oauth.js centralizes the generic pieces: config resolution, PKCE utilities, auth URL generation, JSON fetch helpers, and JWT parsing. New websites can import it instead of rewriting those primitives.

Operational Checklist and Troubleshooting

Before first deployment

Verify the PocketBase schema exists, the admin email/password are correct, and the auth server domain matches the issuer you want to publish.

Before adding a new service

Register the service in oauth_clients, configure its redirect URL exactly, and add its browser origin to ALLOWED_ORIGINS if it calls the auth API from the client side.

If login fails

Check the browser callback URL, the client id, the redirect URI, and whether the PKCE verifier still exists in session storage.

If PocketBase requests fail

Confirm POCKETBASE_URL points to PocketBase itself, not the auth server, and make sure the admin credentials are set in Vercel.

If CORS blocks the browser

Add the exact origin to ALLOWED_ORIGINS. CORS matching here is strict by origin string.

If Vercel changes do not appear

Deploy again after changing environment variables. Vercel only applies new environment values to new deployments.
