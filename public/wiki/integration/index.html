Third-party integration

How to connect any website to Felixx Identity

Use this guide if you want your own website or app to log users in through Felixx Identity. The supported flow is OAuth 2.0 Authorization Code with PKCE. That means the browser starts the sign-in, receives a short-lived authorization code, and exchanges it for tokens.

Auth server https://auth.felixx.dev
Required flow Authorization Code + PKCE
Client setup One OAuth client record per website

OAuth Flow

1. Register your app

Create an OAuth client record in PocketBase with a unique `client_id`, your app name, and the exact redirect URI(s) you will use.

2. Send the user to `/authorize`

Build an authorization URL with `response_type=code`, `client_id`, `redirect_uri`, `state`, `code_challenge`, and `code_challenge_method=S256`.

3. Validate the callback

When the browser returns with `code` and `state`, compare the state to what you saved locally before continuing.

4. Exchange the code

Send the code to `/token` with the PKCE verifier. The server returns access and refresh tokens if everything matches.

5. Use the access token

Send `Authorization: Bearer <token>` to protected account endpoints or to `/userinfo` if you only need identity claims.

6. Rotate or revoke

Refresh tokens are rotated by calling `/token` with `grant_type=refresh_token`, and can be revoked with `/revoke`.

Endpoints

Method Path Use
GET /authorize Start the sign-in flow and render the Felixx login screen.
POST /authorize/login Used by the hosted login page after the user enters credentials.
POST /token Exchange an auth code or refresh a token.
POST /revoke Invalidate a refresh token.
GET /userinfo Fetch the signed-in user profile.
GET /api/account/profile Read the current user profile.
PATCH /api/account/profile Update name, email, or avatar URL.
POST /api/account/password Change the account password.
GET /api/account/apps List connected apps and sessions.
POST /api/account/apps/:clientId/revoke Revoke all sessions for a connected app.
GET /api/account/sessions List active sessions for the current user.
POST /api/account/sessions/:sessionId/logout Log out a specific session.

Browser Apps

Recommended pattern

  1. Store the PKCE verifier and state in sessionStorage.
  2. Store the token response in localStorage only if you accept browser-side token persistence.
  3. Use the access token for API calls.
  4. Refresh when a request returns `401` or the access token expires.
Keep the redirect URI exact. The auth server checks it against the registered OAuth client record.

Server Apps

Recommended pattern for backend-rendered sites

If your site has a server, you can still use the same OAuth flow. Let the browser redirect to Felixx Identity, then handle the callback on your backend or exchange the code on the server after verifying state and PKCE.

This is useful when you want to keep refresh tokens out of browser storage.

Example Snippets

const authBaseUrl = "https://auth.felixx.dev";
const clientId = "your-client-id";
const redirectUri = "https://your-site.example/callback";

// 1. Generate PKCE verifier/challenge
// 2. Save verifier + state locally
// 3. Send the browser to /authorize
// 4. Exchange the returned code at /token
fetch(`${authBaseUrl}/userinfo`, {
  headers: {
    Authorization: `Bearer ${accessToken}`
  }
});
await fetch(`${authBaseUrl}/revoke`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ token: refreshToken })
});

Security Notes

Do

  • Use PKCE for every public client.
  • Use HTTPS redirect URIs only.
  • Register each redirect URI exactly.
  • Store short-lived state locally and clear it after callback.

Don’t

  • Don’t skip the state check.
  • Don’t use the auth server as a generic password API.
  • Don’t share a single OAuth client across unrelated websites.
  • Don’t assume a refresh token is permanent.
If you’re integrating Felixx Identity into your own site, start by registering a client, then copy the browser flow from this page into your app.
